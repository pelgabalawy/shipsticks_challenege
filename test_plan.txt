================================================================================
                        SHIPSTICKS BOOKING FLOW - TEST PLAN
================================================================================

Project:        ShipSticks Web Application
Environment:    Staging (https://app.staging.shipsticks.com)
Test Type:      End-to-End / UI Functional
Flow:           Booking - Happy Path

--------------------------------------------------------------------------------
1. PRECONDITIONS
--------------------------------------------------------------------------------

  1.1  The staging environment (app.staging.shipsticks.com) is up and
       reachable over HTTPS.
  1.2  Test data is defined in config.json: origin/destination addresses,
       shipment type, item, size, service level, and delivery date.
  1.3  The delivery date has available shipping service options for the
       selected route and service level.
  1.4  No user account is required — the test runs as a guest.
  1.5  Playwright browsers are installed (playwright install).

--------------------------------------------------------------------------------
2. TEST STEPS - HAPPY PATH
--------------------------------------------------------------------------------

  Step 1   Navigate to the application URL.
  Step 2   Enter the origin address (autocomplete selection).
  Step 3   Enter the destination address (autocomplete selection).
  Step 4   Select the trip type (One Way / Round Trip).
  Step 5   Click "Get Started."
  Step 6   Verify redirection to the /book/ship page.
  Step 7   Accept the user notification prompt ("I understand").
  Step 8   Add an item to be shipped (e.g., Golf Bag).
  Step 9   Select the item size (e.g., Standard).
  Step 10  Open the delivery date picker.
  Step 11  Navigate to the correct month and select the target date.
  Step 12  Select the service level (e.g., Ground).
  Step 13  Click "Next: Traveler Details" to proceed to Step 2 of booking.

--------------------------------------------------------------------------------
3. EXPECTED RESULTS
--------------------------------------------------------------------------------

  - The user is successfully redirected to /book/ship after clicking
    "Get Started."
  - The selected item and size are reflected correctly in the order summary.
  - The chosen date is consistent across the date picker and the listed
    service options.
  - The user is able to advance to the Traveler Details step without errors.

--------------------------------------------------------------------------------
4. KEY ASSERTIONS
--------------------------------------------------------------------------------

  Step 6   The page URL contains /book/ship after clicking "Get Started."
  Step 9   The selected size radio has aria-checked="true", confirming the
           correct size (e.g., Standard) is active in the HeadlessUI radio group.
  Step 8   The "Item Sizes" section is rendered on the page. This section only
           appears after an item has been added to the cart.
  Step 11  All "Ships on" dates displayed on the service level cards fall
           before the selected delivery date.
  Step 13  The page URL contains /book/login after clicking "Next: Traveler
           Details," and the step progress bar shows a check-circle icon for
           Step 1, confirming it is complete.

--------------------------------------------------------------------------------
5. TOP 3 RISKS
--------------------------------------------------------------------------------

  1. Shipping or delivery dates may not be available for the selected route,
     causing the date picker or service options to return no results.

  2. Service level options (e.g., Ground) may have no availability for the
     given origin/destination and date combination.

  3. Provided addresses may not be recognized by the autocomplete service,
     preventing the booking flow from starting.

  4. The end-to-end test covers the entire Step 1 flow in a single run,
     which increases execution time and flakiness. Breaking it into smaller,
     focused test cases would reduce risk and improve reliability.
